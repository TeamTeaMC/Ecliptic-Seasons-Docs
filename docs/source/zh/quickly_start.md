# Ecliptic Seasons 玩家指南

  *A Player Guide to Seasonal Worlds*

  ------

## 1. 引言

  Minecraft 的世界原本是一个几乎不随时间变化的环境。
  白天与夜晚不断循环，天气偶尔变化，但环境本身几乎不会随着时间推移而产生明显变化。

  Ecliptic Seasons 的目标是改变这一点。

  该模组引入了一套基于 **二十四节气（Solar Terms）** 的季节系统，使世界在时间尺度上呈现出更接近自然环境的变化。随着时间的推移，玩家将逐渐观察到环境的转变：

    - 昼夜长度随季节变化
    - 气候逐渐转冷或转暖
    - 雨雪出现的概率随节气改变
    - 植物与农作物的生长条件发生变化
    - 动物行为与环境氛围随季节调整

  与许多仅提供视觉变化的季节模组不同，Ecliptic Seasons 的设计目标是构建一个 **气候系统（Climate System）**。

  在这一系统中，节气不仅影响环境的外观，还会逐渐改变：

    - 天气模式
    - 农业条件
    - 温度环境
    - 生物行为

  ------

## 2. 节气世界

  Ecliptic Seasons 使用二十四节气作为时间划分方式。
  在这一模型中，世界的时间结构如下：

  ```text

1 Year = 4 Seasons
1 Season = 6 Solar Terms

  ```
  
  默认情况下，每一个节气持续 **7 个 Minecraft 日**。

  因此，一个完整的年度周期为：

  ```text
24 Solar Terms × 7 Days = 168 Days
  ```

这一周期长度可以在配置文件中进行调整，以适应不同服务器或整合包的需求。对于长期运行服务器，推荐设置每个节气持续36~
72天，以适应现实时间，因为每个MC日只有20天。

------

### 昼夜变化

随着季节变化，昼夜长度也会随之改变。

在夏季，白天时间会更长，而在冬季，夜晚则会更长。这一变化是通过动态调整日照时间实现的。

例如：

| 季节 | 白天长度 |
|----|------|
| 夏季 | 更长   |
| 冬季 | 更短   |

------

### 节气提示

当世界进入新的节气时，系统会自动在聊天栏显示提示信息。

这一功能可以帮助玩家了解当前世界所处的时间阶段，例如：

- 春分
- 夏至
- 秋分
- 冬至

玩家可以根据节气信息判断环境变化，例如降雪开始的时间、作物适宜的种植时期等。

------

### 现实时间同步（可选）

Ecliptic Seasons 还提供了一个可选功能，使游戏节气与现实世界日期同步。

当启用该功能后：

  ```text
  现实世界日期 → 对应游戏节气
  ```

例如：

- 现实立春 → 游戏立春
- 现实冬至 → 游戏冬至

这一功能适用于长期服务器或沉浸式玩法，但在某些情况下可能与其他时间相关模组产生冲突，因此默认关闭。

------

## 3. 天气与气候系统

Ecliptic Seasons 的天气系统与原版 Minecraft 存在一个重要区别。

在原版游戏中，天气是 **全局统一的**：
一旦开始降雨，整个世界都会同时进入降雨状态。

Ecliptic Seasons 则采用了 季节性群系气候模型。

------

### 局部天气

由于引入了群系气候系统，在同一时间内，世界不同区域可能出现不同天气。

例如：

| 群系 | 天气  |
|----|-----|
| 平原 | 降雨  |
| 山地 | 降雪  |
| 沙漠 | 无降水 |

因此，玩家可能会观察到以下现象：

- 一侧山地正在降雪
- 附近森林正在降雨
- 沙漠保持晴朗

这种现象是气候系统的一部分，而不是错误。

------

### 降水概率

降水概率会随节气发生变化。

例如，在夏季平原中，降雨可能更加频繁。

这些变化通过配置文件中的降水倍率控制，例如：

  ```text
  RainChancePercentMultiplier
  ThunderChancePercentMultiplier
  ```

需要注意的是，这些倍率只会影响全局倍率，并不会完全决定最终天气。

最终天气仍然会受到：

- 全局倍率

- 群系概率（降雨量/打雷概率）

- 群系数量（这也可能会影响）

的共同影响。

------

### 统一天气（可选）

对于某些服务器或整合包，玩家可能更倾向于传统的“全局天气”模式。

Ecliptic Seasons 提供了一个可选配置，使所有主世界群系共享同一降水事件。

启用以下配置即可实现：

  ```text
  RainTogether = true
  SnowTogether = true
  ```

在这种模式下，天气表现将更加接近原版 Minecraft，这需要群系标签正确，且群系本身需要有降水。

------

### 天气过渡

为了避免天气变化过于突然，Ecliptic Seasons 引入了天气过渡系统。

当玩家在不同气候区域移动时，天气会逐渐过渡，而不是瞬间改变。例如，从晴天进入降雨区域时，降雨会逐渐增强，而不是立即开始。

这一机制通过以下参数控制：

  ```text
  WeatherBufferDistance
  WeatherTransitionSpeed
  ```

这些参数决定了天气过渡的范围和速度。

------

### 天气投票机制

在某些情况下，其他模组可能会绕过 Ecliptic Seasons 的天气 API 直接读取全局降水信息。

为了保持天气逻辑一致，系统提供了一个 **天气投票机制**。

该机制会根据玩家所在位置统计天气状态，并决定全局天气类型。

配置参数：

  ```text
  WeatherVotePercent
  ```

这一机制可以有效避免在复杂模组环境中出现天气逻辑冲突。

------

## 4. 农业与湿度系统

Ecliptic Seasons 对农业系统进行了较大程度的扩展。
在原版 Minecraft 中，大多数农作物只需要满足两个条件即可正常生长：

- 有足够光照
- 种植在合适的方块上

而在 Ecliptic Seasons 中，作物生长还受到 **季节与湿度环境** 的影响。

这一设计旨在模拟真实农业环境，使种植行为更加具有策略性。

------

### 季节性作物

启用季节农业后，农作物不再在所有季节中保持相同生长效率。

不同作物会具有适宜的季节。例如：

| 作物类型 | 适宜季节    |
|------|---------|
| 小麦   | 春季 / 夏季 |
| 南瓜   | 夏季      |
| 甜菜   | 秋季      |

当作物处于不适宜的季节时，可能出现以下情况：

- 生长速度明显降低
- 骨粉无法促进生长
- 完全停止生长

该机制由配置项控制：

  ```
  EnableSeasonalCrop = true
  ```

如果关闭该选项，作物将恢复原版行为。

此外， Ecliptic Seasons 与标准季节模组不同的是，其可以设定不同群系和不同节气的生长概率，例如在春季适宜的作物在立春和春分的生长概率是不同的。

------

### 湿度环境

除了季节之外，作物还受到 **环境湿度** 的影响。

湿度来源包括：

- 附近水体（气泡柱，不稳定加湿）
- 群系气候
- 降雨天气
- 温室结构

湿度不足时，作物可能无法正常生长，即使季节正确。

湿度系统由以下配置项控制：

  ```
  EnableCropHumidityControl = true
  ```

启用后，系统会持续监测区块环境，并计算当前湿度水平。

------

### 湿度过渡

为了避免湿度突然变化导致农业行为不稳定，系统提供了湿度过渡机制。

  ```
  CropHumidityTransition = true
  ```

Ecliptic Seasons的群系湿度受到多种因素影响，但注意这个值存在季节性波动。

这个选项会使得湿度连续变化，而不是离散突变，使得某些群系的种植体验更好。

------

### 骨粉限制

在启用季节农业后，骨粉不再总是成功。

当作物处于错误季节或湿度不足时，骨粉可能会失败。

------

### 强制兼容模式

由于 Minecraft 中存在大量不支持Neo/Forge事件管线的模组作物。

为了解决这一问题，Ecliptic Seasons 提供了强制兼容模式。

  ```
  ForceCompatMode = true
  ```

启用后，所有注册了标签的植物都会遵循基础农业规则。

------

## 5. 温室系统

温室是 Ecliptic Seasons 中的重要农业机制之一。

该系统允许玩家在不适宜的季节或者湿度环境下中种植作物，通过模拟封闭农业环境来改善生长条件。

温室系统由多个要素组成：

- 温室结构
- 季节核心（Season Core）
- 湿度调节装置

------

### 温室检测

温室的核心机制是 **封闭结构检测**。

系统会检查作物周围一定范围内的建筑结构，以判断是否形成温室环境，检测机制很简单，只需要检查是否为封闭空间即可。

检测范围由以下配置控制：

  ```
  GreenHouseMaxDiameter
  GreenHouseMaxHeight
  ```

默认情况下：

| 参数   | 默认值 |
|------|-----|
| 最大直径 | 32  |
| 最大高度 | 10  |

当建筑结构满足封闭条件时，该区域将被视为温室。

------

### 光照限制

温室并不能完全忽略光照条件。

如果温室内部光照过低，大部分作物仍然可能停止生长。

这一概率由配置控制：

  ```
  LowLightGreenhouseFailChance
  ```

数值越高，低光照环境下失败概率越大。

因此，温室仍然需要合理的照明设计。

------

### 简化温室模式

对于不希望复杂农业机制的玩家，系统提供了简化模式。

  ```
  SimpleGreenHouseMode = true
  ```

启用后：

- 不需要特殊核心方块
- 不需要湿度设备
- 只需封闭结构即可

这种模式适合轻量化整合包。

------

### 季节核心

季节核心 是温室系统的核心组件。

它能够在一定范围内提供环境修正效果，例如：

- 改善作物季节条件
- 提供农业增益
- 扩展温室影响范围

影响范围由以下配置控制：

  ```
  SeasonCoreRange = 15
  ```

玩家可以通过合理布局多个核心来覆盖大型农场。

------

### 祈祷仪式

季节核心还支持一种特殊机制：**季节祈祷仪式**。

该仪式可以使得借助无核心的温室容器内孕育出季节核心。

------

### 湿度设备

湿度设备用于调节温室内部环境。

当启用时，可以显著提高作物生长稳定性，可以查看JEI了解。

------

## 6. 温度与生存环境

除了天气变化与农业系统之外，Ecliptic Seasons 还引入了一定程度的 **环境温度影响**。

这一系统的目标并不是完全替代生存类温度模组，而是为世界提供更加合理的环境反馈，使季节变化不仅体现在视觉效果上，也会影响玩家体验。

------

### 季节温度变化

随着季节变化，世界的整体温度会发生改变。

系统并不会直接修改原版生物群系的温度，而是通过附加的季节修正来模拟气候变化。

是否下雪取决于群系温度或者是否有特殊设定。

如果没有设置下雪时间，那么群系下雪时间将随温度分配。

对于偏寒冷的群系如针叶林而言，春季仍然可能下雪。

对于温暖群系如平原，其在冬季的后半段才会下雪。

------

### 季节性冰雪

在寒冷季节中，部分区域可能出现降雪或结冰。

Ecliptic Seasons 默认不开启这个功能，而是使用冬季氛围替代，其具有更好的服务器性能。

配置选项：

  ```
  IceAndSnow = false
  IceAndSnowMelt = false
  ```

当启用这些选项后：

- 冬季更容易出现冰雪
- 温暖季节中冰雪会逐渐融化

------

### 寒冷群系

某些群系被视为 **永久寒冷区域**。

在这些群系中，即使开启了季节性冰雪机制，且处于温暖时节，也可能出现以下情况：

- 雪不会完全融化
- 水仍然可能结冰

相关配置：

  ```
  SnowKeepInSnowyBiomes = true
  ```

这意味着高山、冰原等区域仍然保持寒冷气候。

------

### 热射病机制

在炎热季节中，玩家可能会受到 **热射病（Heatstroke）** 的影响。

当玩家在高温群系中停留过久，尤其是在夏季正午时段，系统可能施加负面效果。

配置项：

  ```
  HeatStroke = true
  ```

该机制鼓励玩家采取一些简单的环境策略，例如：

- 在阴影区域休息
- 避免正午长时间暴露
- 在森林或水边活动

该系统不会造成极端生存压力，而主要用于增强季节氛围。

------

## 7. 环境效果

为了让季节变化更加明显，Ecliptic Seasons 引入了大量环境视觉效果。

这些效果主要是 **装饰性环境反馈**，不会对世界结构造成破坏。

------

### 雪覆盖系统

最显著的变化之一是 **雪覆盖效果**。

在冬季到来时，地表可能逐渐出现一层视觉雪覆盖。

相关配置：

  ```
  SnowyWinter = true
  ```

这一系统与传统雪层不同，它并不会生成真实的雪方块，而是作为一种视觉效果覆盖在地面上。

这种设计具有两个重要优点：

1. 不会改变世界方块结构
2. 减少大规模更新带来的性能问题

因此，玩家无法使用铲子收集这些雪层。

------

### 树木与雪

树木在冬季也会出现雪覆盖效果，特别是每个树叶。

相关配置：

  ```
  SnowyTree = true
  ```

当启用时：

- 树叶表面会出现霜雪效果
- 森林会呈现明显的冬季景观

这一机制主要用于增强视觉氛围。

------

### 光照影响

为了保持自然效果，雪不会在强光附近积累。

配置项：

  ```
  NotSnowyNearGlowingBlock = true
  ```

例如：

- 火把附近
- 萤石附近
- 高光照区域

这些地方通常不会出现雪覆盖。

------

### 草地与树叶颜色变化

随着季节变化，植被颜色也会发生变化。

配置选项：

  ```
  SeasonalGrassColorChange = true
  SeasonalColorChangeExtend = true
  ```

例如：

| 季节 | 植被颜色  |
|----|-------|
| 春季 | 明亮绿色  |
| 夏季 | 深绿色   |
| 秋季 | 偏黄或棕色 |
| 冬季 | 偏灰    |

这种渐变效果可以让世界看起来更加自然。

------

### 春季花朵

在春季，草地上可能出现随机花朵。

配置：

  ```
  FlowerOnGrass = true
  ```

这一效果模拟春夏季草地植物生长的景象。

------

### 季节粒子

模组还加入了多种环境粒子，用于增强氛围。

例如：

- 春季蝴蝶
- 夏季萤火虫
- 秋季落叶
- 候鸟迁徙

相关配置：

  ```
  SeasonalParticles = true
  ```

这些粒子主要用于视觉装饰，不会影响游戏性能。

------

### 自然声音

不同季节也会播放不同的环境声音。

配置：

  ```
  NaturalSound = true
  ```

例如：

- 春季鸟鸣
- 夏季昆虫声
- 秋季风声

这些声音会根据季节自动变化。

------

## 8. Configuration Guide

### 常用配置说明

Ecliptic Seasons 提供了较为丰富的配置选项，使玩家和服务器管理员能够根据需要调整季节系统的行为。

大多数情况下，默认配置已经能够提供稳定且合理的体验。然而，在某些服务器或整合包环境中，适当调整部分参数可以显著改善游戏体验。

下面介绍几个最常被修改的配置选项。

------

### 季节长度

节气持续时间决定了季节变化的速度。

  ```text
  LastingDaysOfEachTerm = 7
  ```

默认情况下：

- 一个节气 = 7 天
- 一年 = 168 天

如果希望季节变化更慢，可以增加该值，例如：

  ```text
  LastingDaysOfEachTerm = 10
  ```

这会使一年延长至 **240 MC日**。

在服务器环境中，较长的季节周期通常更受欢迎，我们推荐一地球日或者两地球日变化一次节气，如设置为72MC日，因为玩家有更多时间体验季节变化。

------

### 初始节气

当新世界创建时，系统会从指定节气开始。

  ```text
  InitialSolarTermIndex = 4
  ```

节气编号范围为：

  ```text
  1 ~ 24
  ```

例如：

| 编号 | 节气 |
|----|----|
| 1  | 立春 |
| 7  | 立夏 |
| 13 | 立秋 |
| 19 | 立冬 |

修改此值可以让新世界从不同季节开始。

------

### 天气同步

默认情况下，天气是 **局部气候系统**。

如果希望天气更加接近原版 Minecraft，可以启用以下配置：

  ```text
  RainTogether = true
  SnowTogether = true
  ```

启用后：

- 所有主世界群系共享同一降水事件
- 世界天气更容易预测

这种模式在某些服务器中更容易管理。

------

### 降雨概率

降雨频率可以通过倍率调整。

  ```text
  RainChancePercentMultiplier = 40
  ThunderChancePercentMultiplier = 20
  ```

数值越高：

- 降雨越频繁
- 雷暴出现概率越高

------

### 季节农业

农业系统可以完全关闭。

  ```text
  EnableSeasonalCrop = true
  ```

关闭后：

- 作物不再受到季节限制
- 游戏体验更接近原版

该选项适合轻量整合包。

------

### 湿度控制

湿度系统控制农业环境。

  ```text
  EnableCropHumidityControl = true
  ```

关闭后：

- 作物不再检查环境湿度
- 农业难度显著降低

建议仅在需要简化玩法时关闭。

------

### 温室简化模式

温室系统可以简化。

  ```text
  SimpleGreenHouseMode = true
  ```

启用后：

- 不需要 Season Core
- 不需要湿度设备
- 只需要封闭结构

这使温室更加容易建造。

------

### 昼夜变化

昼夜长度变化由以下配置控制：

  ```text
  DynamicDaylightDuration = true
  ```

关闭后：

- 昼夜长度恢复原版
- 季节仅影响环境效果

该选项适用于某些技术型整合包。

------

### 粒子与视觉效果

如果玩家遇到性能问题，可以关闭部分视觉效果。

例如：

  ```text
  SeasonalParticles = false
  NaturalSound = false
  ```

这些效果主要用于增强氛围，并不会影响核心玩法。

------

## 9. 从 静谧四季 迁移

许多玩家最初接触季节模组时使用的是 **Serene Seasons**。
Ecliptic Seasons 的设计目标并不是简单替代，而是提供一种更加灵活且可扩展的季节系统。

因此，在迁移整合包时，理解两者之间的差异非常重要。

------

### 核心设计差异

Serene Seasons 的设计更接近传统季节系统。

它主要关注：

- 作物季节
- 降雪行为
- 基础温度变化

而 Ecliptic Seasons 的目标是构建一个 **完整的气候系统**。

主要扩展包括：

- 二十四节气时间体系
- 局部天气系统
- 湿度农业
- 温室机制
- 环境视觉变化

因此，两者在设计理念上存在明显差异。

------

### 雪机制差异

Serene Seasons 使用真实雪层。

这意味着：

- 雪会生成方块
- 世界方块会被修改

Ecliptic Seasons 默认使用 **视觉雪覆盖系统**。

这种机制：

- 不生成真实雪层
- 不修改世界结构

优点包括：

- 更好的性能表现
- 与远距离渲染模组兼容
- 不会留下永久雪层

------

### 天气系统差异

Serene Seasons 使用 **全局天气系统**。

Ecliptic Seasons 默认使用 **局部气候系统**。

这意味着：

- 不同群系可能同时出现不同天气
- 气候变化更加自然

如果玩家更喜欢传统天气，可以启用：

  ```text
  RainTogether = true
  SnowTogether = true
  ```

------

### 作物系统差异

Serene Seasons 主要使用作物标签控制季节。

Ecliptic Seasons 在此基础上增加了：

- 湿度系统
- 温室农业
- 环境修正

因此农业玩法更加复杂，但也更具有策略性。

------

### API 兼容

许多模组仍然使用 Serene Seasons API。

为了兼容这些模组，可以使用 **Serene API 兼容桥**。

该桥接模组会：

- 模拟 Serene Seasons API
- 将季节数据转发到 Ecliptic Seasons

这样可以让原本依赖 Serene Seasons 的模组继续正常工作。

------

### 迁移建议

如果玩家希望从 Serene Seasons 迁移，通常可以按照以下步骤进行：

1. 移除 Serene Seasons
2. 安装 Ecliptic Seasons
3. 根据需要选择是否安装Serene Seasons API Stub (Ecliptic Seasons Bridge)、Ecliptic Seasons: Bundles、Ecliptic Seasons:
   MultiMod Patch
4. 调整农业配置

在大多数情况下，世界存档不会受到影响，因为 Ecliptic Seasons 不会修改原版方块数据。

------

### 何时不建议迁移

在以下情况下，可能需要谨慎迁移：

- 整合包强依赖 Serene Seasons API
- 使用大量基于 Serene Seasons 的农业模组
- 已经对 Serene Seasons 进行了大量配置

在这种情况下，建议测试新世界后再迁移。

------

## 指南总结

Ecliptic Seasons 的核心目标是让 Minecraft 世界拥有更真实的时间与气候变化。

通过引入节气系统、气候模拟与环境反馈，玩家能够体验到一个随着时间不断变化的世界。

无论是：

- 春季花开
- 夏季虫鸣
- 秋季落叶
- 冬季白雪

这些变化都让 Minecraft 的世界变得更加生动。

## Ecliptic Seasons FAQ

### 常见问题解答

------

### 1. 为什么我听到雷声，但没有下雨？

Ecliptic Seasons 使用 **局部天气系统**。

天气是否降水不仅取决于世界天气，还取决于当前所在的群系气候。因此可能出现以下情况：

- 附近群系正在发生雷暴
- 当前群系不满足降水条件

这种情况下，玩家仍然可能听到远处雷声。

如果希望所有群系天气同步，可以启用：

  ```text
  RainTogether = true
  ```

------

### 2. 为什么春天有时候还会下雪？

节气系统并不是简单的四季切换。

在以下情况下仍然可能降雪：

- 群系温度较低
- 高海拔区域
- 寒冷群系

例如：

- 山地
- 冰原
- 冷气候森林

因此在春季早期看到降雪是正常现象。

------

### 3. 为什么作物不生长？

通常有三个原因：

#### 1 季节不适合

某些作物只适合特定季节。

#### 2 湿度不足

作物需要合适的环境湿度。

#### 3 光照不足

尤其是在温室中。

如果骨粉无法生效，通常说明环境条件不满足。

------

### 4. 温室为什么不起作用？

温室系统需要满足一定结构条件。

常见问题包括：

- 温室结构未完全封闭
- 温室范围超过检测尺寸
- 光照不足

默认检测范围：

| 参数 | 默认值 |
|----|-----|
| 直径 | 32  |
| 高度 | 10  |

如果结构过大，部分区域可能不会被识别为温室。

------

### 5. 温室一定需要季节温室核心吗？

不一定。

如果启用：

  ```text
  SimpleGreenHouseMode = true
  ```

温室只需要封闭结构即可。

------

### 6. 为什么有时候冬季下雨但没有雪？

降水类型取决于：

- 群系温度

- 当前节气

如果温度仍然较高，即使是冬季，也可能降雨而不是降雪。

------

### 7. 为什么河流附近天气看起来奇怪？

河流两岸的群系不同，因此天气不同。

------

### 8. 为什么夏天会出现热射病？

在炎热群系中，如果玩家长时间暴露在夏季正午阳光下，可能触发 **Heatstroke**。

该机制可以关闭：

  ```text
  HeatStroke = false
  ```

------

### 9. 为什么某些模组作物没有季节限制？

并非所有模组作物都注册了季节标签。

------

### 10. 为什么天气在移动时会改变？

天气具有 **过渡系统**。

当玩家进入不同气候区域时，天气会逐渐变化。

------

### 11. 为什么整合包推荐使用这个模组？

Ecliptic Seasons 的优势包括：

- 更好的性能
- 不修改世界方块
- 更灵活的配置
- 更复杂的气候系统
- 支持远距离渲染模组

------

### 12. 如何让依赖 Serene Seasons 的模组继续工作？

可以使用 **Serene Seasons API Stub (Ecliptic Seasons Bridge)**。

该模组会：

- 模拟 Serene Seasons API
- 将数据转发到 Ecliptic Seasons

这样许多原本依赖 Serene Seasons 的模组仍然可以运行。

------

### 13. 看起来覆雪模型不支持某些资源包，如FastGrass，会出现z-index问题

这些资源包修改了原版的方块模型面，因此推荐在资源包里打开LegacySnowyBlock。
同时在客户端设置中找到`CullTopFaceWithSnow`进行开启以剔除这些模型的表层面。
